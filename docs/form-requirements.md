# 表單需具備的系統設定

要讓一張 UOF 電子簽核表單能由本 MCP **完整且正確操作**（查欄位 → 起單 → 查詢 → 簽核 →
結案，且結案後的下游處理正常），這張表單在 UOF 後台需要符合以下條件。工具本身已經到位，
能不能用得起來主要取決於表單的設定。

> 本文件是「起單」這條路徑的後台前提。起單由 SOAP/PublicAPI 完成；以下是與機制無關、
> 表單本身需具備的設定。

## 前提清單

| # | 條件 | 如何確認 | 由誰設定 |
|---|---|---|---|
| 1 | 表單已發佈版本（有 `recentVersionId`） | `get_form_list` 回傳可看到版本 | 表單設計者 |
| 2 | 對外的中介欄位涵蓋所有業務必填欄位 | 比對 `get_form_structure_by_id` 與網頁上的必填欄位 | 後台中介欄位設定 |
| 3 | 中介欄位是可儲存值的型別（非佔位欄） | 起單後用 `get_task_result` 撈回，確認值有存進去 | 後台中介欄位設定 |
| 4 | 不依賴明細表格／實際附檔，或這些非必填 | 看欄位結構與必填 | 表單設計者 |
| 5 | 第一站簽核者可決定：自由流程由起單者指定，或已設好固定流程 | 試起單／看流程設定 | 流程設定 |
| 6 | 操作帳號對該表單有申請權限，且已設好部門與職級 | 試起單；缺職級會起單失敗 | 帳號／組織設定 |
| 7 | 結案的下游整合能處理外部起單帶入的資料 | 結案後確認下游模組是否正常 | 下游模組設定 |

其中只有第 1 點可由 API 自動檢核，第 2～7 點屬系統設定層面，需人工或後台確認，工具無法代補。

## 重要觀念

- **API 起單成功 ≠ 表單業務完整，也 ≠ 結案後下游成功。** 工具只能填到「對外中介欄位」，
  且 UOF 不會在外部起單時驗證網頁必填欄位。若中介欄位對應不完整，會建出缺欄位的單；
  某些表單在結案時還會觸發下游模組，若資料不足可能在 API 之後失敗（呼叫端看不到）。
- 因此導入每一張新表單前，請逐項對照上表確認，並在測試環境實際走一次「起單 → 撈回欄位 →
  結案 → 確認下游」再投入正式使用。

## 採購單現況（範例）

採購單已有版本、可起單查詢，但對外中介欄位只開放 4 個（表單編號、主要欄位、備註、附件），
SOAP `SendForm` 無法填網頁上的主旨、供應商、幣別、採購明細等必填欄位。

本 repo 已針對採購單系列登錄 `ops/web_apply` handler：`apply_form` 會透明改以 Playwright 填網頁表單，
可完整填主旨、供應商、幣別、付款條件、儲存地點、運送方式、到貨日與明細。若未來新增其他客製 plugin
表單，仍需逐單補 web handler，或在 UOF 後台補齊中介欄位後走 SOAP。
