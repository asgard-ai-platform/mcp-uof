# 表單需具備的系統設定

一張 UOF 電子簽核表單能否由本 MCP 完整操作，同時取決於表單設定、目前解析器涵蓋度與該帳號權限。

> 本文件是「起單」這條路徑的後台前提。起單一律以 httpx 網頁機制完成（沒有 SOAP、沒有其他機制可選）；以下是與機制無關、表單本身需具備的業務/流程設定。

## 前提清單

| # | 條件 | 如何確認 | 由誰設定 |
| --- | --- | --- | --- |
| 1 | 表單已發佈版本（有 `recentVersionId`） | `get_form_list` 回傳可看到版本 | 表單設計者 |
| 2 | 若欄位由客製 dialog/plugin 驅動，相關控制項與 postback 配方須已支援 | 比對結構工具輸出與 UOF UI，並在測試環境寫入後回讀 | 開發面／部署端私有 skill |
| 3 | 一般網頁解析（`_parse_field_blocks`）掃到的欄位涵蓋業務必填欄位，且是可寫入的實際輸入元件（非唯讀佔位欄） | 比對 `get_form_structure_by_id` 與網頁上的必填欄位；起單後用 `get_task_result` 撈回確認值有存進去 | 表單設計者（欄位版面）；解析缺漏屬開發面 bug |
| 4 | dataGrid 明細的欄位對應與 row-editor postback 須已支援 | 看欄位結構是否標 dataGrid，起單後回讀確認明細落地 | 開發面 |
| 5 | 流程已在 UOF 後台設定完成；目前 MCP 起單不會套用 `first_signer_account` | 試起單並查詢實際簽核路徑 | 流程設定；需指定首簽者時使用 Web UI |
| 6 | 操作帳號對該表單有申請權限，且已設好部門與職級 | 試起單；缺職級會起單失敗 | 帳號／組織設定 |
| 7 | 結案的下游整合能處理外部起單帶入的資料 | 結案後確認下游模組是否正常 | 下游模組設定 |

第 1、5、6、7 點是 UOF 業務/組織層面的既有前提，與實作機制無關。第 2、3、4 點是 httpx 解析／填單配方
的涵蓋度問題。能否操作取決於目前解析器是否看得到控制項，以及對應的 postback 配方是否已實作。
沒打通就是待補的開發缺口，不是系統對外開放範圍的天花板。

## 重要觀念

- **起單成功 ≠ 表單業務完整，也 ≠ 結案後下游成功。** `apply_form` 會驗證解析器辨識到的必填欄位、選項與明細，但無法驗證未解析欄位或站台內部的所有業務規則。
  若某張表單的欄位涵蓋度不完整，仍可能建出缺欄位的單。
  某些表單在結案時還會觸發下游模組，若資料不足可能在 API 之後失敗（呼叫端看不到）。
- 因此導入每一張新表單前，請逐項對照上表確認，並在測試環境實際走一次「起單 → 撈回欄位 → 結案 → 確認下游」再投入正式使用。**欄位涵蓋度不完整時，先假設是解析器/配方待補的 bug 去查根因，不要當成表單本身「就是只能填這些」而放棄追查。**

## 特殊 plugin 表單

有些表單本體是 JS 對話框驅動的 plugin：一般網頁解析（`_parse_field_blocks`）可能只掃到少數容器欄位，
真正的主旨、供應商、員工、明細等藏在對話框與回填機制裡。這是 DOM 渲染方式造成的，不是後台刻意少開放。

本 repo 不內建這類表單的逐張 SOP。部署端應以不納入版本控制的私有 skill 定義欄位語意、操作順序與業務驗證。
